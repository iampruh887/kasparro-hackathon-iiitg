# Agentic Architecture Overview

The `Agentic Representation Optimizer` uses an orchestration architecture that coordinates specialized "agents" to ingest, process, evaluate, and synthesize merchant data. This deterministic, multi-stage pipeline is defined in `src/agenticAnalyzer.js` and ensures that data flows logically and reliably toward producing a final readiness assessment.

## Pipeline Stages

The architecture now operates in `agentic-verified-loop` mode. Each stage corresponds to an individual agent:

### 1. Intake Agent (`runIntakeAgent`)
**Purpose:** Ensure standard payload normalization.
**Action:** Processes the incoming raw data (whether JSON payload or scraped HTML), validating and normalizing core fields: `storeProfile`, `products`, `policies`, `faqs`, `reviews`, `trustSignals`, and `structuredData`.

### 2. Signal Extraction Agent (`runSignalExtractionAgent`)
**Purpose:** Map the volumetric spread and completeness vector.
**Action:** Measures the scale of incoming data (product count, policy count, structured flags). It outputs a `completenessVector` indicating whether critical aspects like "catalog," "policy," "FAQ," "socialProof," and "machineReadable" data are `present` or `missing`.

### 3. Risk Scoring Agent (Wrapper around `analyzeStore`)
**Purpose:** Generate deterministic heuristic scores.
**Action:** Evaluates the normalized payload across five critical categories: Product Clarity, Policy Coverage, FAQ Coverage, Trust Signals, and Technical Structured Data. It evaluates representation gaps and produces a comprehensive analysis object with severity-ranked issues (Critical, High, Medium, Low).

### 4. Verification Agent (`runVerificationAgent`)
**Purpose:** Validate analysis integrity before final recommendations are accepted.
**Action:** Checks score-range validity, issue-to-action coverage, action ordering consistency, and contradiction context quality. Produces a verification quality score and pass/fail result.

### 5. Refinement Agent (`runRefinementAgent`)
**Purpose:** Repair malformed analysis output when verification fails.
**Action:** Sanitizes issue fields, normalizes numeric bounds, recomputes priority scores, and rebuilds ranked actions. This enables iterative correction loops.

### 6. Prioritization Agent (`runPrioritizationAgent`)
**Purpose:** Strategy plotting.
**Action:** Consumes the list of prioritized actions generated by the scoring stage. It buckets these recommendations into `quickWins` (effort < 25) vs `strategicBets` (effort > 25) to provide actionable roadmaps to merchants.

### 7. Consistency Agent (`runConsistencyAgent`)
**Purpose:** Execution stability & confidence determination.
**Action:** Evaluates the number of `critical`, `high`, and `contradictionCount` flaws to assign an overall algorithmic confidence score and band (`high`, `medium`, `low`). This score reflects how reliable the AI considers its own analysis of the store given the quality of input.

### 8. Synthesis Agent (`runSynthesisAgent`)
**Purpose:** Executive summary generation.
**Action:** Aggregates findings from scoring, extracting, and prioritizing stages to compose a human-readable `executiveSummary`, ensuring all parts of the pipeline are connected effectively.

## Orchestration & Tracking

The primary orchestrator (`agenticAnalyzeStore`) coordinates these agents.

- **Parallel Execution:** `runSignalExtractionAgent` and the `RiskScoringAgent` execute in parallel via `Promise.all` to optimize initial ingestion performance.
- **Closed Loop Verification:** The orchestrator runs an iterative loop (`verify -> refine -> re-verify`) for up to a maximum loop count. Loop trace and verification checks are returned in `agentic.verification`.
- **Stage Timings:** Under `run.stageTimingsMs`, each agent logs execution duration for debugging and performance tracking.
- **Objective Mapping:** The orchestrator converts outcomes into unified metrics via `deriveAnalysisObjectives`, tracking continuous goals like *Catalog Coverage*, *Trust Stability*, *Machine-Readability*, and *Decision Stability*.

## Integration With Local LLM
After the deterministic pipeline evaluates the structural risk, the backend sends the `executiveSummary` strings, issues, and `completenessVector` to a local `Ollama` model (`gemma4:latest` with fallback to `gemma3:latest`, managed by `src/ollama.js`). The local LLM generates conversational and contextual insights (Narrative, Merchant Reply, Trust Assessment) based on the strict architectural findings.
